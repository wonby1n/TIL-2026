# Jenkins · Docker · Nginx는 각각 뭐 하는 애들인가?

- Docker: 실행 환경을 통째로 포장하는 애
- Jenkins: 빌드·테스트·배포를 자동으로 시키는 애
- Nginx: 들어오는 트래픽을 받아서 적절한 서비스로 보내주는 애

---

## 1️⃣ Docker — “환경을 통째로 고정시키는 도구”

### Docker가 해결하는 문제
> “내 컴퓨터에서는 되는데 서버에서는 안 돼요”

### Docker가 하는 일
- 애플리케이션 + 실행 환경(OS, 라이브러리, 설정)을 하나의 이미지로 만듦
- 어디서 실행해도 항상 동일한 결과 보장

### 핵심 개념

- 이미지(Image): 실행 설명서(설계도)
- 컨테이너(Container): 이미지로부터 실행된 실제 프로세스
- Dockerfile: 이미지를 만드는 레시피

### 실무에서 Docker의 역할
- 로컬 개발 환경 표준화
- 서버 배포 단위
- CI/CD(Jenkins 등)의 실행 기반

### 👉 Docker 없으면
- 서버마다 설치 방법 다름
- 팀원 환경 차이로 오류 폭발

---

## 2️⃣ Jenkins — “사람 대신 배포 버튼 눌러주는 자동화 로봇”

### Jenkins가 해결하는 문제
> “매번 빌드하고 서버 접속해서 배포하기 귀찮고 실수도 많음”

### Jenkins가 하는 일
- Git에 코드가 올라오면:
    1. 코드 가져오기
    2. 테스트 실행
    3. 빌드
    4. Docker 이미지 생성
    5. 서버에 배포

        → 이걸 자동으로

### 핵심 개념

- CI (Continuous Integration)
    - 코드 합칠 때마다 자동 테스트/빌드
- CD (Continuous Deployment)
    - 통과하면 자동 배포

### Jenkins가 직접 “서비스를 돌리진 않음”
- ❌ 웹서버 아님
- ❌ API 서버 아님
- ⭕ 배포 파이프라인 관리자
- 👉 보통 Jenkins는 Docker를 실행하는 쪽임
- (“도커를 굴리는 관리자”)

--- 

## 3️⃣ Nginx — “문지기 + 교통정리 담당”

### Nginx가 해결하는 문제
> “요청이 어디로 가야 하는지, 어떻게 처리해야 하는지”

### Nginx가 하는 일
1. 리버스 프록시
    - /api → Spring Boot
    - / → Vue
2. HTTPS 처리
    - SSL 인증서 관리
3. 부하 분산
    - 요청을 여러 서버로 분산
4. 정적 파일 서빙
    - Vue 빌드 결과물 서빙

### 핵심 포인트
- 브라우저는 Nginx만 바라봄
- 실제 서비스 위치는 숨김

👉 보안 + 구조 안정성 때문에 거의 필수

---

## 4️⃣ 셋이 같이 쓰일 때 구조 (중요)

```scss
[사용자 브라우저]
        ↓
     Nginx (80/443)
        ↓
 ┌───────────────┐
 │ Spring Boot   │
 │ Docker        │
 └───────────────┘
        ↓
    Postgres

[ Jenkins ]
   ↑
 Git push
```

### 흐름 설명
1. 개발자가 Git에 push
2. Jenkins가 감지
3. Jenkins가 Docker 이미지 빌드
4. 서버에 배포
5. Nginx가 트래픽을 Spring/Vue로 라우팅


## 5️⃣ 이 셋의 “역할 차이” 표로 정리


| 구분 | Docker | Jenkins | Nginx |
| --- | --- | --- | --- |
| 역할 | 실행 환경 | 자동화 | 트래픽 제어 |
| 서비스 실행 | ⭕ | ❌ | ❌ |
| 배포 자동화 | ❌ | ⭕ |	❌ |
| 네트워크 처리 | ❌ | ❌ | ⭕ |
| 필수 여부 | 거의 필수 | 선택 | 거의 필수 |
