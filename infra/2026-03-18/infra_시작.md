## 0단계. 먼저 뭘 받는지부터 이해하기

EC2 접속하려면 보통 3개가 필요해.

### 1) 서버 주소

예:

```bash
3.39.123.45
```

이건 인터넷에 있는 내 서버의 주소야.

### 2) pem 키 파일

예:

```bash
mykey.pem
```

이건 **서버 문 여는 열쇠**야.
비밀번호 대신 쓰는 파일이라고 생각하면 돼.

중요:

* 잃어버리면 곤란함
* 남한테 주면 안 됨
* 권한 설정을 해야 할 수도 있음

### 3) 접속 계정명

보통 이런 것 중 하나야.

* `ubuntu`
* `ec2-user`
* `root`는 보통 잘 안 씀

SSAFY에서 우분투 서버를 주면 보통 `ubuntu`일 가능성이 큼.

## 1단계. SSH로 EC2 접속하기

이게 제일 첫 관문이야 🔥

### 명령어 형태

```bash
ssh -i mykey.pem ubuntu@3.39.123.45
```

### 뜻 해석

```bash
ssh                  # 서버에 접속할게
-i mykey.pem         # 이 pem 키 파일을 사용할게
ubuntu@3.39.123.45   # ubuntu 사용자로 저 서버에 들어갈게
```

### 접속 전 체크할 것

#### 1) 현재 터미널 위치 확인

pem 파일이 있는 폴더에서 실행해야 편해.

```bash
pwd                  # 현재 위치 보기
ls                   # 현재 폴더 안 파일 보기
```

`ls` 했을 때 `mykey.pem` 이 보여야 해.

#### 2) pem 권한 변경

리눅스/WSL/우분투/맥에서는 보통 이걸 해줘야 할 때가 많아.

```bash
chmod 400 mykey.pem
```

뜻:

```bash
chmod 400 mykey.pem   # key 파일을 나만 읽을 수 있게 권한 제한
```

왜 하냐면 SSH는 pem 파일 권한이 너무 열려 있으면 보안상 위험하다고 보고 접속을 막는 경우가 많아.

### 실제 접속

```bash
ssh -i mykey.pem ubuntu@3.39.123.45
```

처음 접속하면 이런 비슷한 문구가 나올 수 있어.

```bash
Are you sure you want to continue connecting (yes/no/[fingerprint])?
```

이건 **"이 서버 처음 접속하는데 믿고 들어갈래?"** 라는 뜻이니까 `yes` 입력하면 돼.

### 접속 성공하면?

이제 내 컴퓨터가 아니라 **서버 안으로 들어간 상태**야.

프롬프트가 이런 느낌으로 바뀔 수 있어.

```bash
ubuntu@ip-172-31-xx-xx:~$
```

이제부터 치는 명령어는 **EC2 서버가 실행**하는 거야.

## 2단계. 서버 안에 들어가서 제일 먼저 할 것

접속했다고 바로 뭔가 설치하지 말고, 먼저 **지금 내가 어디 있고 서버 안이 어떻게 생겼는지** 확인해.

### 자주 쓰는 기본 명령어

```bash
pwd                  # 현재 위치 확인
ls                   # 현재 폴더 안 목록 보기
cd                    # 폴더 이동
mkdir project         # project 폴더 만들기
```

예시:

```bash
pwd
ls
mkdir myproject
cd myproject
pwd
```

뜻:

* 지금 어디 있는지 보고
* 뭐가 들어있는지 보고
* 새 폴더 만들고
* 그 폴더 안으로 들어가기

---

## 3단계. 프로젝트 코드를 서버에 올리기

이제 서버에 **내 프로젝트 코드**가 있어야 실행 가능해.

보통 방법은 2가지야.

### 방법 1) git clone

이게 제일 많이 씀.

```bash
git clone 저장소주소
```

예:

```bash
git clone https://github.com/example/project.git
```

그럼 깃허브에 있는 프로젝트가 서버에 복사돼.

그리고:

```bash
cd project
ls
```

### 방법 2) 직접 파일 업로드

이건 가능하긴 한데 보통 협업 프로젝트면 `git clone` 이 더 일반적이야.

초보 기준에서는
**"코드는 보통 git clone으로 서버에 가져온다"**
이렇게 기억하면 돼.

---

## 4단계. 환경변수(.env) 넣기

이제 서버에 코드는 생겼는데, 프로그램이 돌아가려면 **설정값**이 필요할 수 있어.

예:

* DB 주소
* DB 비밀번호
* API 키
* JWT secret
* 포트 번호

이런 건 보통 `.env` 파일에 적어.

### 예시

```bash
nano .env
```

그러면 편집기가 열리고 이런 식으로 적을 수 있어.

```env
DB_HOST=localhost
DB_PORT=3306
DB_USER=root
DB_PASSWORD=1234
OPENAI_API_KEY=xxxx
PORT=8080
```

저장 후 종료.

### nano에서 자주 쓰는 것

* 수정하고
* `Ctrl + O` → 저장
* 엔터
* `Ctrl + X` → 종료

### 확인

```bash
cat .env
```

뜻:

```bash
cat .env              # .env 파일 내용 확인
```

주의: 실제 비밀번호나 키가 있으면 팀원 공유 화면이나 발표 중엔 함부로 보여주면 안 돼.

---

## 5단계. 프로젝트 실행하기

이건 프로젝트 언어에 따라 달라져.
즉, 여기서는 **확실하지 않음**이 있어.
왜냐하면 Java인지, Python인지, Node인지에 따라 실행 명령어가 다르기 때문이야.

그래서 대표적으로 나눠서 볼게.

### 경우 1) Python 프로젝트

```bash
python app.py
```

또는

```bash
uvicorn main:app --host 0.0.0.0 --port 8000
```

### 경우 2) Node.js 프로젝트

```bash
npm install
npm run dev
```

또는

```bash
npm run start
```

### 경우 3) Java / Spring Boot 프로젝트

```bash
./gradlew bootRun
```

또는

```bash
java -jar app.jar
```

### 여기서 중요한 개념 하나

프로젝트를 실행하면 프로그램이 **특정 포트에서 기다리기 시작**해.

예:

* `3000`
* `8080`
* `8000`

즉:

> "누가 이 포트로 들어오면 내가 응답할게"

이 상태가 되는 거야.

---

## 6단계. 포트 개념 다시 연결하기

예를 들어 백엔드가 `8080`에서 실행됐다면, 서버 주소가 `3.39.123.45` 라고 할 때 접속 주소는 이런 느낌이야.

```text
http://3.39.123.45:8080
```

뜻:

* `3.39.123.45` → 서버 주소
* `8080` → 그 서버 안에서 백엔드가 열려 있는 문 번호

---

## 7단계. 왜 실행은 됐는데 접속이 안 될 수 있나?

초보가 여기서 진짜 많이 막혀 😢
근데 이건 정상이다.

접속 안 되는 대표 이유:

### 1) 프로그램이 죽었음

실행한 줄 알았는데 꺼졌을 수 있음.

확인:

```bash
ps
top
htop
```

### 2) 포트가 외부에 안 열려 있음

AWS 보안그룹이나 서버 방화벽에서 포트를 막았을 수 있어.

예:

* `22`는 SSH용
* `8080`은 백엔드용
* `3000`은 프론트용

즉, 서버 안에서 실행돼도 밖에서 들어오는 문을 안 열면 접속 안 돼.

### 3) localhost로만 실행됨

이건 진짜 자주 나오는 문제야.

예를 들어 앱이 `127.0.0.1` 또는 `localhost` 에만 묶여 있으면 서버 내부에서만 보이고, 외부에서는 안 보일 수 있어.

그래서 어떤 프레임워크는 실행할 때 이렇게 해야 해.

```bash
--host 0.0.0.0
```

이건
**"외부에서도 들어오게 열어둘게"**
라는 느낌이야.

## 8단계. 서버 상태 확인할 때 자주 쓰는 명령어

### 현재 위치 / 파일 보기

```bash
pwd                  # 현재 위치 보기
ls                   # 파일 목록 보기
```

### 파일 확인 / 수정

```bash
cat .env             # 파일 내용 보기
nano .env            # 파일 수정하기
```

### 권한 변경

```bash
chmod 400 mykey.pem  # pem 키 권한 변경
```

### 실행 중인 프로그램 보기

```bash
ps                   # 프로세스 확인
top                  # 실시간 상태 보기
htop                 # 보기 편한 상태 보기
```

### 강제 종료

```bash
kill PID             # 특정 프로세스 종료
```

여기서 `PID`는 프로세스 번호야.

## 9단계. 인프라 초보가 실제로 머릿속에 넣어야 하는 흐름

이건 진짜 중요해서 아주 짧게 다시 정리할게.

```text
내 컴퓨터
→ ssh로 ec2 접속
→ 서버 안에서 프로젝트 코드 받기
→ .env 설정하기
→ 프로젝트 실행하기
→ 포트 확인하기
→ 외부 접속 확인하기
```

대장님이 지금 이 흐름만 머리에 넣으면 돼.

## 10단계. 아주 간단한 예시 시나리오

예를 들어 백엔드 프로젝트를 EC2에 올린다고 해보자.

### 순서

```bash
# 1. 내 컴퓨터에서 서버 접속
ssh -i mykey.pem ubuntu@3.39.123.45

# 2. 서버 안에서 프로젝트 받기
git clone https://github.com/example/project.git

# 3. 프로젝트 폴더로 이동
cd project

# 4. 환경변수 작성
nano .env

# 5. 실행
npm install
npm run start
```

그리고 앱이 `8080`에서 열렸다면:

```text
http://3.39.123.45:8080
```

으로 접속 시도해보는 거야.

## 11단계. 지금 당장 외워야 하는 것 5개

1. **EC2는 인터넷에 있는 내 서버다**
2. **SSH는 그 서버에 들어가는 방법이다**
3. **.env는 비밀값/설정값을 넣는 파일이다**
4. **포트는 서버 안 프로그램의 문 번호다**
5. **프로그램이 실행돼도 포트나 설정이 막혀 있으면 접속이 안 된다**

---

## 12단계. 다음으로 바로 배워야 할 것

이제 다음 순서가 좋아.

1. `ssh` 접속 실습
2. `.env` 파일 만들기
3. 프로젝트 실행 명령어 이해
4. `ps`, `kill`, `top` 으로 상태 확인
5. `Nginx` 배우기

왜냐하면 Nginx는 이제 막 실행된 앱을 **80번 포트(웹 기본 주소)**와 연결해줄 때 많이 나오기 때문이야.
